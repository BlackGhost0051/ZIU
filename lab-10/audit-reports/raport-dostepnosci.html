4. Test czytnika ekranu (NVDA / VoiceOver) — 3 obserwacje

Obserwacja 1 — Linki "Kliknij tutaj"
NVDA odczytuje listę linków jako trzy identyczne "Kliknij tutaj". Użytkownik nie jest w stanie odróżnić linków bez kontekstu wizualnego. Lista linków (Insert+F7) pokazuje trzy identyczne pozycje.
Kryterium: 2.4.4 A • Priorytet: P1
Obserwacja 2 — Przycisk emoji ❤️
NVDA odczytuje przycisk jako "serce button" (dosłowny opis emoji przez czytnik). Użytkownik nie wie, że przycisk służy do dodania do ulubionych.
Kryterium: 4.1.2 A • Priorytet: P1
Obserwacja 3 — Formularz bez etykiet
VoiceOver odczytuje pole input jako "edit text" bez żadnej nazwy. Nie wiadomo, co należy wpisać. Pola z tabindex 1/2/3 są odwiedzane w kolejności 1→2→3, zamiast kolejności wizualnej na ekranie.
Kryterium: 1.3.1 / 2.4.3 A • Priorytet: P1

7. Podsumowanie i rekomendacje

7.1 Ogólna ocena dostępności

Stan dostępności audytowanej strony oceniamy jako krytyczny. Wynik Lighthouse 57/100 przy 18 naruszeniach WCAG 2.1 (15× poziom A, 3× poziom AA) oznacza, że strona jest praktycznie niedostępna dla użytkowników korzystających z czytników ekranu i nawigacji klawiaturą. Aż 10 błędów priorytet P1 blokuje podstawowe funkcje: wypełnienie formularza, kliknięcie przycisku CTA, zrozumienie treści obrazów i linków. Narzędzia automatyczne wykryły 20–50% wszystkich problemów — potwierdza to konieczność obowiązkowego testowania manualnego jako stałego elementu procesu wydawniczego.

7.3 Rekomendacje

Rekomendacja 1 — Kontrast i design system
Wprowadzić design tokens z zatwierdzonymi parami kolorów spełniającymi WCAG 1.4.3: np. tekst podstawowy #111111 na białym (18:1), akcent #0043FF z białym tekstem (5.9:1). Zintegrować eslint-plugin-jsx-a11y w pipeline CI/CD — wykrywa brak alt, label, aria-label już na etapie code review, zanim błąd trafi na produkcję.
Rekomendacja 2 — Semantyczny HTML zamiast divów
Każdy interaktywny element musi być natywnym elementem HTML: <button> zamiast <div onclick>, <a href> dla linków nawigacyjnych. Stosować landmark'i: <main>, <nav>, <header>, <footer>. Nigdy nie używać dodatnich wartości tabindex — DOM powinien odzwierciedlać kolejność wizualną. Zasada: jeśli element jest klikalny, musi być <button> lub <a>.
Rekomendacja 3 — Dostępność jako etap procesu deweloperskiego
Obecny proces nie zawiera żadnego kroku weryfikacji dostępności — błędy są wykrywane dopiero po wdrożeniu. Proponowany proces: (1) axe-core lub pa11y-ci w GitHub Actions blokuje merge PR przy nowych naruszeniach WCAG2AA; (2) testy manualne klawiaturą przed każdym release'm; (3) cykliczny audyt z NVDA/VoiceOver raz na kwartał; (4) szkolenie zespołu z podstaw WCAG — 2h wystarczą do wyeliminowania 80% najczęstszych błędów.

7.4 Wnioski dotyczące procesu

Audyt potwierdził, że automatyczne narzędzia (Lighthouse, axe, pa11y) wykrywają razem maksymalnie 50% błędów — w tym przypadku 8–13 z 18, czyli 44–72%. Błędy takie jak ukryty focus (#3), pusty alt na obrazie informatywnym (#7), nieopiszące linki (#8) czy brak fieldset (#16) zostały wykryte wyłącznie testami manualnymi. Oznacza to, że żadne pojedyncze narzędzie nie zastąpi kompletnego audytu. Optymalny zestaw dla projektu komercyjnego to: axe w CI/CD (automatyczna bariera) + Lighthouse w DevTools (szybki przegląd) + pa11y w raporcie (dowód dla klienta) + test NVDA/VoiceOver przed release'm (rzeczywiste doświadczenie użytkownika).
